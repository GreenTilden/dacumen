Generalizing the n=3 instances of the same shape the team has now hit:

1. **GOV-06 L04** — `append-only-ingester-stale-mapping`. `git_events.deployment_id` snapshotted at backfill time. When `PROJECT_ENDPOINTS` later gained a `deployment_id` for a repo, every prior event for that repo kept `deployment_id: null`. `evidence_coverage` drifted down silently as the config improved. Fix: `remap_git_events()` + `POST /api/reconciliation/remap` (idempotent fill-only refresh).

2. **GOV-08 L01** — `suggestions[].deployment_name` snapshotted at suggestion creation (`reconciler.py:329`). The nodemad deployment was renamed twice since 2026-04 (`nodemad` → `nodemad — N0D3MAD-01 living portable DArnTech platform` → `nodemad — NODEMAD-03 voice-capture-appliance + fleet-template`) and the dashboard's `group-by-deployment_name` showed phantom buckets — three names for one `deployment_id: c6fa7aa0`, with the current name appearing in zero suggestions. Fix: reader-side resolution at `GET /suggestions` + `GET /suggestions/summary` (replace stored name with live lookup) + `POST /api/reconciliation/refresh-deployment-names` (idempotent backfill of the 7 stale rows; 80 unchanged; 0 orphans).

3. **The clean shape — the activity log**: `log_reconciliation_activity()` (`reconciler.py:398`) stores **only** `deployment_id`. When a consumer needs the deployment's current name, it joins live against `deployments.json`. This is the design that DOESN'T have the bug — no snapshot to go stale. It's the affirmative half of the lesson, not just "what to fix" but "what shape was right all along."

**Why:** Snapshotting display labels feels like a small optimization at write-time but creates a permanent reconciliation debt: every rename / config change / restructure of the parent now requires a backfill, and you have no guarantee the backfill ever runs. The asymmetry is brutal — the *write* costs one extra field, the *forever* costs a backfill pass plus a reader fix per consumer that ever looked at the stale column. Worse, the staleness is silent: the data isn't *wrong* in any single row, it's wrong only when compared to current truth, which means it slips past row-level audits. The activity log shape (foreign key only, name joined live) trades a tiny per-read cost for never having this bug. The reconciler shape (snapshot at write-time) was a premature optimization that paid for itself in GOV-06 + GOV-08 cleanup.

**How to apply:** When adding a new table/document that stores a row referencing a parent entity (deployment, project, customer, repo, tenant, sprint), **only store the foreign-key id**. For display, join live against the parent at read time. If a snapshot is genuinely necessary (caching for performance, audit-snapshotting for historical accuracy on a specific timestamp), do it deliberately — name the field something that announces its snapshot semantics (`deployment_name_at_creation`, `customer_name_snapshot_2026_q1`), and pair every snapshot field with an idempotent backfill endpoint from day one (don't ship the write side without the refresh side). If you inherit a system with snapshotted denormalized labels (you'll find them by greping for `*_name` next to a `*_id` of the same entity), apply the GOV-08 pair: reader-side resolution at every consumer + an idempotent backfill that fills-only (never invents a name for a missing parent, never overwrites a non-null with null).
