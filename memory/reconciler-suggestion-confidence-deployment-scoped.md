Surfaced 2026-05-15 during the GOV-07 post-close walk of 87 pending reconciliation suggestions. Lorna Financials' `REQ-DEPLOYMENT-001` ("System may be packaged as a darntech-freelance-ops open-source template with sanitized config, example data, and setup documentation for solo freelancer financial...") was scored `confidence: high, 0.9` and proposed `→ implemented` — but the evidence chain was the previous day's `RC6: rag-core-client rename — requirements + Dockerfile + import` commit. The open-source template hasn't shipped; the rename has nothing to do with template packaging. The high-confidence score came purely from "this commit happened on this deployment, and this requirement is pending on this deployment." Dismissed with reason captured in the suggestions audit trail.

DArnometer's 23 high-conf suggestions in the same walk were all genuinely valid — but that's *because* DArnometer's evidence chains (Vue charts, weather edges, baseball backtest commits) happened to semantically match the pending step/req statements. The reconciler didn't verify the match; it just rode the deployment-scope coincidence. With less-narratively-coherent activity (Lorna had the RC6 rename, which is cross-cutting infrastructure work that touched every consumer repo simultaneously) the false-positive shape surfaces.

**Why:** A `high, 0.9` confidence label is operator instruction — it tells the human "you can batch-approve these without close reading." When the score is structurally generous (deployment-scoped, not content-aware), the operator either learns to distrust the label (defeats the purpose) or batch-approves false positives (worse). This is the same shape as [[fix-without-action-surface-reconciliation]] applied to the confidence surface: a UI signal that doesn't honestly reflect the underlying mechanism. Operator-of-the-moment can't tell from the label which 0.9s are content-grounded and which are deployment-coincidence. Specifically dangerous when cross-cutting renames (RC6, rag-core-client) deploy across every consumer repo — every pending suggestion on every consumer deployment will get an evidence chain that day, regardless of relevance.

**How to apply:** When reviewing reconciliation suggestions, do not trust the confidence label as a proxy for evidence relevance — read at least one evidence summary against the target_statement and spot-check for keyword/semantic overlap. The label tells you the reconciler *thinks* it's confident, not that the match is content-grounded. If the next GOV sweep wants to address this structurally, the fix lives in `casey-junior/app/services/reconciler.py`: confidence should derate when the target_statement's distinctive keywords don't appear in the evidence summaries. (Distinctive keywords meaning content nouns specific to the requirement — "open-source template", "gate registry", "kelly bankroll" — not generic ones like "endpoint" or "service".) Until that's structural, the operator review process is the only guard.
